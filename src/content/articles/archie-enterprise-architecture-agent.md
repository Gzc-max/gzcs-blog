最近做了一个比较完整的 AI 全栈项目：**Archie 企业架构智能助理**。

这个项目不是一个简单的 ChatGPT 壳子，而是围绕企业架构管理场景，把 **Agent 对话、动态技能、架构图生成、僵尸应用分析、附件理解、用户权限和审计日志** 串成了一套可落地的业务系统。

从简历角度看，它的价值不在于“接了一个大模型接口”，而在于：我把大模型能力放进了真实业务闭环里，并处理了工程化系统真正会遇到的状态管理、权限边界、工具编排、流式响应、数据持久化和可维护性问题。

## 项目一句话介绍

Archie 是一个面向企业架构管理场景的智能体平台，用户可以通过自然语言完成应用分析、架构上下文图生成、僵尸应用识别、报告生成和技能扩展等工作。

系统采用 **FastAPI + LangGraph/LangChain + React/Vite + SQLAlchemy** 的全栈架构，后端提供 Agent 编排、工具调用、技能管理、会话存储和 RBAC 权限能力，前端提供对话式工作台、技能中心、右侧视图面板、用户/角色管理和审计日志。

## 为什么这个项目值得写进简历

很多 AI 项目停留在“输入问题，返回回答”，但真实业务系统需要解决的问题更多：

- 用户是否有权限使用某个功能？
- 大模型什么时候该调用工具，什么时候只回答文本？
- 工具执行过程如何通过流式状态反馈给前端？
- 生成的架构图如何在当前会话中识别、打开、恢复和复用？
- 上传的附件如何解析成 Agent 可理解的上下文？
- 技能如何动态启停，而不是写死在代码里？
- 管理员如何追踪用户登录和 Agent 提问记录？

Archie 的实现重点就是把这些问题做成一个完整闭环。

## 技术架构总览

项目整体分为后端服务和前端工作台两部分：

```text
archie
├── src                    # FastAPI 后端
│   ├── main.py             # HTTP 入口、Agent 运行、SSE、技能、附件、会话接口
│   ├── agents              # Archie Agent 构建
│   ├── skills              # 动态技能注册、加载、中间件
│   ├── tools               # ES 查询、僵尸应用分析、报告生成、上下文图工具
│   ├── modules             # 认证、用户、角色、权限、审计、附件模块
│   ├── storage             # 数据库、Memory、S3 存储
│   └── integrations         # EAM、QuickEA 等外部系统集成
│
└── Archie (1)              # React/Vite 前端
    ├── src/App.tsx          # 对话工作台、技能中心、视图联动
    ├── src/api.ts           # API 封装、Token 刷新、错误处理
    ├── src/auth             # 登录态与权限上下文
    ├── src/pages            # 登录、用户管理、角色管理、审计日志
    └── src/components       # 业务组件
```

后端核心是 `GraphService`：它根据当前运行模式构建 Agent 或 Workflow，接收用户输入，准备会话摘要和附件上下文，然后通过 LangGraph 执行，并把运行过程以普通响应或 SSE 流式响应返回给前端。

前端核心是对话式工作台：左侧承载会话、消息、技能入口和附件；右侧承载架构图、HTML artifact、历史视图和详情面板。这样用户不是只拿到一段文本，而是在同一个界面完成“提问 - 分析 - 生成视图 - 继续追问 - 保存资产”的连续工作流。

## 核心设计一：动态技能系统

Archie 没有把所有工具固定写进 Agent，而是抽象出一套技能系统。

一次请求中，前端会带上当前启用的 `active_skills`。后端根据这些技能 ID 构建技能包，再把技能对应的工具和中间件装配到 Agent 中。

这个设计带来几个好处：

- **能力可插拔**：不同业务场景可以启用不同技能，比如僵尸应用分析、上下文图生成、报告生成。
- **上下文更干净**：没有启用的技能不会干扰本轮推理，减少无关工具选择。
- **易扩展**：后续新增技能时，不需要重写 Agent 主流程。
- **可管理**：前端技能中心可以做启用、停用、上传和删除。

从工程角度看，这比“给 Agent 塞一堆工具”更接近真实平台的形态。

## 核心设计二：企业架构上下文图生成

项目里有一个很有代表性的能力：用户通过自然语言要求“画某个系统的上下文关系图”，Agent 会调用后端工具，通过外部平台 API 生成可预览的架构图。

这里踩过一个关键设计点：**前端不能依赖临时的 tool event 作为唯一数据源**。

因为流式回答和历史消息恢复时，最稳定的数据其实是 assistant 最终输出的 Markdown 文本。所以项目采用了这样的方案：

- 工具在最终回答中输出稳定字段，比如视图名称、视图 ID、预览地址和隐藏的 `diagram_base_url`。
- 前端从 assistant 消息内容中解析这些字段。
- 当前会话中识别到新视图时，自动打开右侧面板。
- 切换会话或刷新历史消息时，也能重新从 Markdown 中恢复视图按钮和右侧面板。

这个方案的优点是：即使中间流式事件没有持久化，最终用户可见的消息也能成为可恢复的状态来源。

这也是我在这个项目里很喜欢的一点：它不是追求“看起来高级”的事件链路，而是优先保证历史可恢复、状态可解释、用户体验稳定。

## 核心设计三：僵尸应用分析闭环

另一个业务亮点是僵尸应用分析。

系统会从 Elasticsearch 查询应用系统指标，然后通过工具链完成评分、风险分级和报告生成。评分维度包括：

- 近 30 天日均活跃用户数
- 近 30 天业务交易次数
- 距最近部署天数

综合评分规则类似：

```text
综合得分 = 0.5 * 用户活跃度得分
        + 0.3 * 业务交易得分
        + 0.2 * 系统维护得分
```

然后按得分划分为高风险僵尸应用、中风险僵尸应用、低风险观察应用和正常应用。

这类能力很适合企业架构管理场景，因为它不是单纯让模型“凭感觉分析”，而是把规则计算、数据查询和自然语言总结结合起来：

- 数据来自真实指标系统；
- 评分逻辑由 Python 工具确定，避免模型自由发挥；
- 模型负责解释结果、组织报告和给出建议；
- 前端以表格、摘要和对话方式呈现结果。

这也是 AI 应用落地里很重要的一条经验：**确定性的部分交给代码，不确定性的表达和归纳交给模型**。

## 核心设计四：附件理解与会话记忆

Archie 支持用户上传附件，并把附件内容绑定到会话中。后端会根据文件类型解析内容，生成摘要、提取文本，并在用户提问时构造附件上下文注入 Agent。

同时，项目还做了会话级记忆：

- 使用 LangGraph checkpointer 保存多轮对话状态；
- 对历史会话生成摘要；
- 构建 Agent 时把会话摘要拼入 system prompt；
- 消息窗口做长度控制，避免上下文无限膨胀。

这个设计解决了两个问题：

- 用户可以围绕一个问题持续追问，不需要每轮重新描述背景；
- 长会话不会无限塞入全部历史，成本和性能更可控。

## 核心设计五：登录、RBAC 和审计

为了让系统更像一个真实内部平台，项目补齐了认证和管理后台：

- 本地账号密码登录；
- JWT Access Token + Refresh Token；
- 前端自动刷新 Token；
- 用户管理、角色管理、权限管理；
- 管理页面和接口级权限校验；
- 登录日志和 Agent 提问日志。

前端通过 `AuthContext` 维护登录态和权限信息，后端通过 FastAPI dependency 做接口鉴权。这样页面按钮显隐只是体验优化，真正的安全边界仍然在后端。

这部分在简历里很有用，因为它说明项目不是 demo，而是具备企业应用该有的用户体系、权限体系和审计能力。

## 我在项目中的技术收获

这个项目给我的最大收获，是对 AI 应用工程化有了更具体的判断。

第一，Agent 不是万能大脑，而是调度层。  
它适合理解用户意图、选择工具、整合结果，但数据查询、规则计算、文件解析、权限判断都应该由确定性代码完成。

第二，流式体验不只是“字一个个出来”。  
真正有价值的流式体验，是让用户知道系统正在做什么：解析附件、准备上下文、调用工具、生成报告、创建视图。这会显著降低等待焦虑。

第三，AI 产品要重视状态恢复。  
只在当前运行时可见的状态不够，刷新页面、切换会话、历史回放时仍然能恢复视图和结果，才是稳定产品体验。

第四，权限和审计不是锦上添花。  
一旦系统进入企业场景，谁能用、谁用了、问了什么、是否越权，都会变成必须回答的问题。

## 可以放进简历的项目描述

如果把 Archie 写进简历，我会这样概括：

> **Archie 企业架构智能体平台**：基于 FastAPI、LangGraph/LangChain、React/Vite 和 SQLAlchemy 构建企业架构智能助理，支持自然语言驱动的架构图生成、僵尸应用分析、报告生成、附件理解、动态技能管理、会话记忆、RBAC 权限控制和审计日志，形成从对话交互到工具执行再到可视化产出的完整 AI 工作流。

可以拆成以下亮点：

- **负责 AI Agent 后端架构设计**：基于 LangGraph/LangChain 封装 Agent 构建、工具调用、会话记忆、流式运行和错误分类，支持同步调用与 SSE 流式响应。
- **设计动态技能系统**：实现技能注册、加载、启停、上传和工具装配机制，使 Agent 能力按业务场景动态扩展，降低工具耦合。
- **落地企业架构图生成链路**：通过自然语言识别上下文图意图，调用外部架构平台 API 生成可预览视图，并在前端实现消息按钮、右侧面板、历史视图恢复等交互。
- **实现僵尸应用分析能力**：接入 Elasticsearch 指标数据，设计加权评分和风险分级规则，自动生成结构化分析报告，提升应用治理效率。
- **搭建完整权限与审计体系**：实现 JWT 登录、Refresh Token、用户/角色/权限管理、接口级鉴权、登录日志和 Agent 提问日志，满足企业内部平台基础治理要求。
- **完善前端对话式工作台**：基于 React/Vite 构建会话、技能中心、附件上传、视图面板、管理后台等页面，支持 AI 结果的持续交互和资产化沉淀。

如果需要更偏招聘 JD 的写法，也可以压缩成一条：

> 基于 FastAPI + LangGraph + React 从 0 到 1 搭建企业架构智能体平台，设计动态技能系统和工具编排机制，接入 ES、对象存储和外部架构平台，实现自然语言生成架构图、僵尸应用分析、附件上下文理解、SSE 流式响应、会话记忆、RBAC 权限和审计日志等能力，推动 AI Agent 从聊天 Demo 落地为可运营的企业级应用。

## 总结

Archie 这个项目让我更明确了一件事：AI 全栈项目的竞争力，不在于“能不能调通模型”，而在于能不能把模型放进真实业务流程里。

一个可落地的智能体平台，至少要处理好几件事：

- 模型和工具的边界；
- 对话和业务状态的持久化；
- 流式交互和前端体验；
- 权限、审计和管理后台；
- 可扩展的技能体系；
- 真实数据和确定性规则。

当这些能力组合起来，AI 才不只是一个回答问题的窗口，而是一个能进入业务系统、协同完成任务的工作台。
