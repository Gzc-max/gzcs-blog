这两天继续看 AI 开发工具方向的更新，我感觉一个变化已经越来越明显：

**coding agent 不再只是帮你补一段代码，而是开始自己把任务推进到 PR 阶段。**

这个变化听起来很顺。

毕竟过去一年我们已经习惯了：

- 让 AI 写函数；
- 让 AI 补测试；
- 让 AI 改样式；
- 让 AI 解释一段老代码；

但最近几条动态连在一起看，事情其实往前走了一步。

- GitHub Copilot coding agent 已经在强调异步完成任务、创建 PR、做自检和安全扫描；
- Google I/O 2026 上，Antigravity 2.0 把 agent-first 开发平台、CLI 和企业工作流继续往前推；
- Codex 这类工具也越来越强调开发者可以在不同入口里分派任务、查看结果、批准或继续迭代。

如果说上一阶段的重点是“AI 能不能写代码”，那下一阶段更现实的问题可能会变成：

**当 AI 把代码、测试结果、PR 描述都端到你面前时，你还会不会审？**

我觉得这件事对前端工程师尤其重要。

因为前端的很多质量问题，并不会在 diff 里一眼露出来。它们藏在状态切换、浏览器运行时、用户路径、样式边界、接口回退和产品语义里。

AI 可以更快地产出改动，但最后能不能合进去，仍然取决于人类有没有能力判断：这是不是一次可靠的交付。

## 为什么“AI 自己开 PR”是一个关键节点

以前我们用 AI 写代码，大多数时候还是在当前编辑器里发生。

人类在主导上下文：

- 我知道自己要改哪个文件；
- 我知道为什么要这么改；
- 我看着 AI 生成结果；
- 我不满意就继续追问；
- 最后我自己整理提交。

这个流程里，AI 像一个能力很强的协作者，但它还没有真正进入团队协作系统。

PR 不一样。

PR 是工程团队里非常特殊的节点。它不是“代码写完了”的证明，而是一次改动进入公共协作区的入口。

一个 PR 至少同时承担几件事：

- 解释问题背景；
- 描述改动范围；
- 暴露风险和取舍；
- 连接测试和验证证据；
- 接受同事审查；
- 最后进入可追溯的历史记录。

所以当 agent 开始自己创建 PR，意义就不是“自动化多了一步”。

它意味着 AI 产出的内容开始进入工程组织的责任链路。

这也是我觉得开发者需要认真看这件事的原因。

只要 PR 还存在，最后的关键能力就不会只是“谁生成得快”，而是：

**谁能把一次改动审清楚。**

## 前端审查的难点，会被 AI 放大

前端 code review 本来就不简单。

后端很多改动可以相对清晰地落在接口、数据结构、事务、权限、性能这些点上。不是说后端容易，而是它的很多风险更容易被测试和日志表达出来。

前端的麻烦在于，它经常同时跨好几层：

- 代码逻辑是否正确；
- 交互状态是否完整；
- 视觉布局是否稳定；
- 移动端是否正常；
- 数据为空、慢、错、重复时是否还能解释；
- 用户是否真的理解这个功能；
- 某个很小的样式改动是否影响了别的页面；

AI 生成前端代码时，很容易“表面完成”。

它能补上组件，能写出状态，能让类型过掉，甚至能写一段看起来挺像回事的 PR 描述。

但很多问题不会自动浮出来：

- 表单禁用态漏了一种边界；
- loading 和 error 的优先级写反了；
- 响应式布局在窄屏挤在一起；
- 组件复用了错误的业务字段；
- 列表刷新后本地选择态没有被清理；
- 文案看起来顺，但和真实业务语义不一致；
- 测试只覆盖 happy path，真正容易坏的路径没碰到；

这些问题过去人类自己写代码时也会发生。

区别是，AI 会把产出速度拉高。

一旦速度上来，审查压力就会跟着上来。

如果团队没有升级 review 方法，最后可能会出现一个很尴尬的局面：

**AI 负责加速生产，人类负责被动兜底。**

这不是一个健康的协作模式。

## 以后看 AI PR，不能只看 diff

我现在越来越觉得，AI 时代的 code review 要从“看 diff”升级成“看证据链”。

只看 diff，会有几个问题。

第一，AI 很会把 diff 写得像合理方案。

它生成的命名通常不差，结构也能模仿项目风格，PR 描述还能主动总结“做了什么”。这会降低人的警惕性。

第二，前端很多风险不在单个文件里。

一个组件改动可能影响路由、状态缓存、埋点、权限、样式继承和接口契约。diff 里每一处都看着没问题，放到真实页面才会出事。

第三，AI 可能会补出“看起来存在”的验证。

比如它说“已添加测试”，但测试可能只是覆盖了组件能渲染；它说“已处理错误态”，但实际只处理了接口 reject，没有处理业务返回空数据；它说“已优化性能”，但只是加了一个记忆化包裹，没有证明瓶颈真的在这里。

所以我觉得之后审 AI PR，至少要多问 5 个问题。

## 1. 这次改动的用户路径是什么？

不要先陷在代码里。

先把问题拉回产品场景：

- 用户从哪里进入；
- 做了什么操作；
- 成功时看到什么；
- 失败时看到什么；
- 中途取消、刷新、切换页面会怎样；
- 移动端或低网速下体验是否还能成立；

AI 很容易从技术任务出发，把需求拆成文件修改。

但前端交付最后面对的是用户路径。

如果 reviewer 自己说不清用户路径，就很难判断这个 PR 到底做对没有。

这也是为什么我不太喜欢只让 AI 给“改动摘要”。

改动摘要只能说明它动了哪里，不能说明它是否完成了用户目标。

一个更好的 AI PR 描述，应该包含用户路径和验证路径。

比如：

- 从列表页进入详情页；
- 修改筛选条件后，列表、URL 和分页状态同步更新；
- 接口失败时保留旧数据并展示错误提示；
- 已用某某数据场景验证空态、慢接口和窄屏布局；

这种描述才值得审。

## 2. 这次改动有没有越过原本的业务边界？

AI 的另一个常见问题，是边界感不稳定。

它可能为了让当前任务通过，顺手改了一个共享组件；也可能为了修一个页面 bug，改了全局 hook；还可能为了补类型，把某个业务字段放宽成 `any` 或可选。

这些改动有时候不是错，但一定要被看见。

前端仓库里最危险的改动，往往不是大张旗鼓的新功能，而是看起来很小的共享层修改：

- 改了基础组件默认行为；
- 改了请求封装的错误处理；
- 改了全局状态初始化逻辑；
- 改了样式变量或布局容器；
- 改了类型定义，让坏数据更容易混进来；

人类写代码时，团队可能会天然知道这些地方敏感。

AI 不一定知道。

所以审查 AI PR 时，我会特别关注“它为了完成任务，动了哪些本不该动的东西”。

如果共享层确实需要改，PR 里就应该给出理由、影响面和验证证据。

如果只是为了图方便，那就应该退回去。

## 3. 验证证据是不是能支撑结论？

以后 AI PR 里最有价值的内容，可能不是那段写得很漂亮的 summary，而是 verification。

但 verification 也要审。

不是出现了“测试通过”四个字就算数。

我更关心这些细节：

- 跑了什么命令；
- 这些命令覆盖了什么层级；
- 有没有浏览器运行时验证；
- 有没有关键截图或 trace；
- 有没有失败过，失败原因是什么；
- 有没有明确说明哪些风险没有覆盖；

尤其是前端改动，我会天然警惕只跑静态检查就宣布完成。

`typecheck` 很重要，`lint` 也很重要，但它们不能证明页面体验正确。

如果这个 PR 改的是交互、布局、状态流或性能路径，验证证据里最好能看到真实运行结果。

这不一定每次都要很重。

小改动可以是截图和关键路径手动验证；中等改动可以加组件测试或 Playwright 冒烟；核心路径改动就应该有更稳定的 E2E 或回归检查。

重点不是工具多炫，而是证据和风险要匹配。

## 4. AI 有没有把“不确定”伪装成“已完成”？

这是我最近很在意的一点。

AI PR 最大的问题之一，不是它会犯错，而是它有时会把不确定的东西写得很确定。

比如：

- “应该不会影响其他页面”；
- “已兼容移动端”；
- “保留了原有行为”；
- “修复了所有边界情况”；

这些话如果没有证据，就是噪音。

人类 reviewer 要训练自己对这类句子敏感。

好的工程协作里，不确定性应该被暴露，而不是被语气抹平。

我更希望看到 AI 在 PR 里写：

- “未覆盖移动端横屏场景”；
- “没有找到现成的 E2E 入口，只做了本地浏览器验证”；
- “该共享组件还被 A/B/C 页面使用，建议重点复核这些路径”；
- “接口返回字段缺少历史样例，当前按类型定义处理”；

这种描述看起来没有那么自信，但反而更可信。

因为它给了 reviewer 真正可行动的信息。

## 5. 这次改动以后，谁能负责回滚和追责？

这听起来有点严肃，但工程系统最终一定会回到责任问题。

AI 可以开 PR，但它不能替团队承担线上事故。

所以每个 AI PR 都应该能回答：

- 这个改动由谁发起；
- 谁确认了需求；
- 谁审过关键风险；
- 合并前有哪些验证证据；
- 出问题时如何快速回滚；
- 哪些监控或日志能证明问题范围；

这不是为了甩锅，而是为了让团队保持工程秩序。

如果 AI 让改动成本大幅降低，而责任链路没有同步升级，团队很快会陷入一种奇怪状态：

产出变多了，但没人真正知道哪些改动是可信的。

这对前端尤其危险。

因为前端问题经常不是一次性崩掉，而是慢慢在用户体验里渗出来：

- 某些用户路径变难用了；
- 某些边界状态开始闪烁；
- 某些页面性能逐渐下降；
- 某个关键按钮在特殊权限下不可见；

等问题被发现时，可能已经过了好几个 AI PR。

所以，review 不只是合并前的礼貌动作。

它是责任链路的一部分。

## 前端团队可以怎么调整自己的 PR 习惯

我觉得不需要一上来搞一套特别重的流程。

真正有用的变化，往往是把几个小习惯变成团队默认值。

## 1. 给 AI PR 一个固定模板

AI 可以自由生成内容，但团队不应该自由接收。

至少可以要求它填这些信息：

- 任务背景；
- 用户路径；
- 改动范围；
- 影响到的共享模块；
- 验证命令；
- 浏览器验证结果；
- 未覆盖风险；
- 回滚方式；

这会强迫 agent 把“我做了什么”转换成“你该怎么审”。

这一步非常关键。

因为 reviewer 最怕的不是代码复杂，而是上下文缺失。

上下文一缺，人就会退回到只看 diff。

而只看 diff，在 AI 产出越来越多的时候会越来越不够。

## 2. 把“可审查性”当成前端架构指标

以前我们评价前端架构，常说可维护、可复用、可扩展。

以后我觉得还要加一个词：

**可审查。**

什么叫可审查？

我的理解是：

- 一个改动的影响边界容易看出来；
- 组件职责清楚，不需要读半天副作用；
- 业务状态有明确来源和流向；
- 共享模块的默认行为有文档或测试兜底；
- 关键路径有能快速跑起来的验证入口；

如果一个页面逻辑全堆在一起，状态从四面八方来，样式靠全局覆盖，接口异常全靠组件里零散判断，那么 AI 改起来会难，人审起来更难。

这种代码在人类时代已经难维护，在 AI 时代会更快暴露问题。

所以我觉得，AI 不是让架构不重要了，而是让架构的反馈周期变短了。

坏边界会被更快踩穿。

## 3. reviewer 要从“挑错”变成“判断风险”

传统 code review 里，很多人会把注意力放在局部问题上：

- 命名能不能再清楚一点；
- 这一行能不能少写一点；
- 这里是不是可以抽函数；

这些当然有价值。

但面对 AI PR，我觉得 reviewer 的优先级要调整。

更重要的是判断风险：

- 这个方案是不是改对了问题；
- 是否破坏了原有行为；
- 是否扩大了改动范围；
- 验证是否覆盖关键路径；
- 失败后是否可回滚；

小的代码风格问题，可以交给格式化、lint、自动建议。

人类 reviewer 更应该把注意力放在 AI 目前还不稳定的地方：

产品语义、业务边界、系统影响、验证质量和责任判断。

这不是说以后不用看代码细节。

而是说，代码细节不再是 review 的全部。

## 4. 不要让 AI 自检替代人类判断

GitHub Copilot coding agent 做自检、安全扫描，Antigravity 强调 agent 工作流编排，这些都很有价值。

但自检不能替代审查。

原因很简单：

AI 自检本质上还是在它自己的理解框架里检查自己。

它可以发现明显遗漏，也可以补一些机械问题，但它不天然知道你的团队真正看重什么：

- 哪些历史包袱不能碰；
- 哪些交互是产品经理反复确认过的；
- 哪些页面虽然代码老，但线上流量很高；
- 哪些“看似合理”的重构会破坏测试同事的验证习惯；

这些信息很多时候存在于团队上下文里。

所以我对 AI 自检的态度是：

**把它当第一层过滤，不要当最终结论。**

它可以帮你减少低级问题，但不能帮你承担工程判断。

## 最后：AI 越会写，人越要会审

我不太认同一种说法：AI 会写代码之后，工程师就只需要提需求。

至少在可预见的一段时间里，我觉得恰恰相反。

AI 越能独立推进任务，工程师越需要具备更强的审查能力。

因为你面对的不再是一段孤立代码，而是一整次由 agent 组织出来的交付：

- 它理解了什么需求；
- 它选择了什么方案；
- 它改了哪些边界；
- 它验证了哪些路径；
- 它隐瞒或遗漏了哪些不确定性；
- 它是否给团队留下了可追溯的证据；

这些问题，才是 AI 开始自己开 PR 之后真正重要的东西。

对前端工程师来说，这可能也是一次职业能力的迁移。

过去我们证明自己，很大一部分靠“我能把功能写出来”。

以后这个能力仍然重要，但它会被放进一个更大的框架里：

**我能不能把 AI、人、代码、测试、浏览器运行时和产品语义，组织成一次可信的交付。**

这听起来没有“十倍工程师”那么刺激，但我觉得更接近真实工作。

真正稀缺的不是让代码更多地出现。

而是让越来越多由 AI 参与生成的代码，仍然能够被理解、被验证、被负责地合并。

这就是下一阶段前端工程师要重新练习的基本功。
